# Architecture Decision Records

> Decisions made during development of Copperline.
> Format: context → decision → consequences.

---

## ADR-001 — Database Schema Grouping

**Date:** 2026-03-28

**Decision:** Use three SQL Server schemas: `Ledger`, `App`, `Mapping`.

| Schema | Tables | Purpose |
|---|---|---|
| `Ledger` | Accounts, Transactions, ImportBatches | Financial records from the bank |
| `App` | Categories, Budgets | User-managed configuration |
| `Mapping` | MerchantRules | Auto-categorisation rules |

**Consequences:** Clean separation of concerns. Queries are self-documenting — `Ledger.Transactions` vs `App.Categories` makes the domain obvious at a glance.

---

## ADR-002 — Amounts Stored as Pence (Integer)

**Date:** 2026-03-28

**Decision:** All monetary values stored as `INT` in pence (e.g. £12.50 → `1250`). Single signed column: positive = credit, negative = debit.

**Consequences:** Eliminates floating-point rounding errors entirely. `SUM(AmountPence)` for any period gives exact balance. Conversion to £ happens only at display time in the frontend.

## ADR-003 — Import Deduplication via SHA256 Hash

**Date:** 2026-03-28

**Decision:** `Ledger.Transactions.ImportHash` is a `CHAR(64)` SHA256 hex string generated by the parser from `AccountId + TransactionDate + Description + AmountPence`. A unique constraint prevents duplicate rows.

**Consequences:** Re-importing the same statement is safe — duplicates are rejected silently. Known edge case: two genuinely identical transactions on the same day (same merchant, same amount) will collide. Accepted as a rare, tolerable limitation.

## ADR-004 — pyodbc over SQLAlchemy

**Date:** 2026-03-28

**Decision:** Use pyodbc with raw T-SQL rather than SQLAlchemy ORM.

**Consequences:** Leverages Steven's existing SQL Server / T-SQL skills directly. No ORM abstraction layer to learn or work around. Queries are explicit and readable. Trade-off: more boilerplate for CRUD operations — acceptable for this project size.

## ADR-005 — SQL Server Instance

**Date:** 2026-03-28

**Decision:** SQL Server instance is `localhost\MSSQL`. Connection uses ODBC Driver 17 with Windows Authentication (`Trusted_Connection=yes`).

**Consequences:** Connection string default in `backend/app/db.py` reflects this. Can be overridden via the `COPPERLINE_DB` environment variable if the instance ever changes.

---

## ADR-006 — Lloyds CSV as Primary Import Format

**Date:** 2026-03-28

**Decision:** Phase 1 import will target Lloyds CSV format only. QIF format ignored for now.

**Key parser requirements:** strip leading `'` from Sort Code; combine split Debit/Credit columns into single signed `AmountPence`; parse dates as `DD/MM/YYYY` explicitly; use `LIKE 'PATTERN%'` prefix matching for merchant rules due to ~18 char description truncation.

**Consequences:** Parser is specific to Lloyds. Designed to be configurable (column mapping by name, not position) so other bank formats can be added later without a rewrite.
